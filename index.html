Dimension 1 - Safety & Regulatory Compliance

ISO 26262, SOTIF, UNECE WP.29 traceability requirements

Q1 · Interpretability
Is the ability to explain and trace system decisions a hard requirement?
Regulatory standards like ISO 26262 and UNECE WP.29 require demonstrable fault traceability. Court or liability contexts may also demand explainable decisions.
Q2 · Failure Analysis
How important is the ability to isolate and debug failure modes in production?
Modular systems allow per-component debugging. End-to-end systems require activation analysis (Grad-CAM, saliency maps) which is less precise.
Q3 · Redundancy
Does the system require independent redundant checks before actuation?
Automotive safety integrity levels (ASIL-D) often mandate that no single point of failure can cause a hazard. Modular systems can insert watchdog checks between components.
🌍

Dimension 2 - Operating Environment

Scenario structure, domain variability, and edge-case frequency

Q4 · Environment Structure
How structured and predictable is the primary operating environment?
Structured environments (highways, airport runways, factory floors) suit rule-based modular logic. Unstructured environments (city intersections, construction zones) benefit from learned flexibility.
Q5 · Edge-Case Frequency
How often will the system encounter rare or out-of-distribution scenarios?
Modular systems struggle to generalize beyond programmed rules. End-to-end and hybrid systems can generalize if trained on diverse data.
Q6 · Multi-Agent Density
What is the density and unpredictability of other agents (pedestrians, cyclists, vehicles)?
High multi-agent density requires sophisticated prediction. End-to-end models can implicitly encode agent interactions; modular systems need explicit prediction modules.
🗄️

Dimension 3 - Data & Computational Resources

Training data availability, labeling budget, and compute constraints

Q7 · Labeled Data Availability
How much labeled training data is available for the target domain?
E2E imitation learning demands large, diverse, and well-labeled datasets. Modular systems can leverage pre-trained components and transfer learning more efficiently.
Q8 · Simulation Access
Is high-fidelity simulation (e.g. CARLA, LGSVL) available for training?
RL-based and E2E methods depend heavily on simulation for safe exploration. Without simulation, RL is impractical. Modular systems are less reliant on simulation.
Q9 · Compute Budget
What is the available onboard compute for inference?
E2E models (especially transformer-based) require significant GPU/NPU resources at inference. Lightweight modular systems can run on embedded processors (NVIDIA Jetson, Mobileye EyeQ).
🚀

Dimension 4 - Deployment & Operational Context

Development maturity, update frequency, and team expertise

Q10 · Development Stage
What is the current development stage of the system?
Research and proof-of-concept stages allow more experimental architectures. Production systems require battle-tested, certifiable approaches.
Q11 · Update Cadence
How frequently will the system need to be updated or retrained?
E2E systems require full retraining cycles. Modular systems allow targeted component updates (e.g., swap perception model) without full revalidation of the stack.
Q12 · Team Expertise
What is the primary expertise of the engineering team?
Modular systems map well to traditional robotics/controls expertise. E2E systems require deep learning and MLOps infrastructure. Hybrid requires both.

Framework Reference - Architecture Properties

Quick-reference comparison across all four paradigms

MOD Modular Architecture
  • Full interpretability per module
  • ISO 26262 / SOTIF compatible
  • Independent component updates
  • Embedded compute friendly
  • Leverages domain expertise
  • Error propagation across modules
  • Limited generalization
  • Complex inter-module interfaces
  • Brittle in unstructured scenes
E2E End-to-End (IL)
  • Simple unified pipeline
  • Strong generalization with data
  • No handcrafted feature engineering
  • Scales with dataset size
  • Black-box, low interpretability
  • Covariate shift vulnerability
  • Requires massive labeled datasets
  • Difficult to certify
  • Overfitting in sparse data regimes
RL Reinforcement Learning
  • No labeled human data required
  • Online self-improvement
  • Discovers non-human strategies
  • Well-suited to continuous control
  • Requires high-fidelity simulation
  • Reward function design is non-trivial
  • Large sim-to-real gap
  • Sample inefficient training
  • Unsafe exploration in real-world
HYB Hybrid Architecture
  • Interpretable perception + planning
  • Adaptive data-driven path planning
  • Balance of safety & flexibility
  • Validated in CARLA 2023
  • Higher integration complexity
  • Requires cross-functional team
  • Interface design non-trivial
  • Larger codebase to maintain
